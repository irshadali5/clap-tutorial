Chapter 0 — Introduction

clap: Command Line Argument Parser

clap is the de-facto standard crate for building CLIs in Rust. It parses arguments, options, flags, and subcommands from std::env::args(), generates help/version output automatically, and validates input — all with zero boilerplate when you use the derive macro.

This tutorial teaches from first principles. Every concept builds on the previous one. You will not need external docs for the core 95% of clap usage.

Prerequisites

You need Rust installed (rustup), basic familiarity with structs and enums, and the ability to run cargo new. That's it.

↯ Mental model — what clap does
$ your-cli --name irshad -v 3 search foo
--name irshad
-v 3
search
foo
→
name : String
  "irshad"
verbosity : u8
  3
command : Commands
  Search { query: "foo" }
Chapter 1

Installation

Add clap to your Cargo.toml. Always enable the derive feature — it's how you'll write clap in 2024.

toml Cargo.toml 
[dependencies]
clap = { version = "4", features = ["derive"] }
Feature flags

You can also add "env" (for reading env vars), "wrap_help" (word-wraps help text), and "unicode". We'll cover these when we need them.

Verify everything compiles:

$ cargo build
Compiling clap v4.x.x
Compiling your-cli v0.1.0
Finished dev [unoptimized + debuginfo] target(s) in 3.2s
Chapter 2

Your First clap Program

The derive approach: you define a struct and annotate it with #[derive(Parser)]. clap reads those annotations and builds the entire parser from your struct's shape.

rust src/main.rs 
use clap::Parser;

#[derive(Parser, Debug)]
#[command(name = "greet", about = "A greeting CLI")]
struct Cli {
    /// Name of the person to greet
    name: String,
}

fn main() {
    let args = Cli::parse();
    println!("Hello, {}!", args.name);
}
$ cargo run -- irshad
Hello, irshad!

$ cargo run -- --help
A greeting CLI

Usage: greet <NAME>

Arguments:
<NAME> Name of the person to greet

Options:
-h, --help Print help
-V, --version Print version
Key insight

The /// doc comment on each field becomes the help text automatically. This is not magic — clap's proc macro reads it at compile time. Always write doc comments on your fields.

What just happened?

Three things the macro wired up for you:

You wroteclap did
struct Cli { name: String }Registered a positional argument called NAME
#[command(about = "...")]Set the help description line
Cli::parse()Parsed argv, validated types, printed errors & exited on failure
Chapter 3

Derive API vs Builder API

clap has two APIs. Always use Derive unless you have a compelling reason not to.

✓ Derive API (recommended)
rust
#[derive(Parser)]
struct Cli {
  /// Port number
  #[arg(short, long, default_value_t = 8080)]
  port: u16,
}
Builder API (verbose)
rust
Command::new("app")
  .arg(Arg::new("port")
    .short('p')
    .long("port")
    .default_value("8080")
    .value_parser(
      clap_value_parser!(u16)))

The Builder API gives you more runtime flexibility (dynamic commands) but most CLIs don't need that. Stick with Derive.

Chapter 4

Arguments, Options, and Flags

These three are the core building blocks of any CLI. The difference is how they're spelled on the command line.

KindExample on CLIHow it looks in struct
Positional argument myapp foo.txt Plain field, no #[arg] needed
Option (named, takes value) myapp --output out.txt #[arg(short, long)] on non-bool field
Flag (named, no value) myapp --verbose #[arg(short, long)] on a bool field
rust src/main.rs — all three kinds 
use clap::Parser;
use std::path::PathBuf;

#[derive(Parser, Debug)]
#[command(about = "File processor")]
struct Cli {
    /// Input file (positional argument)
    input: PathBuf,

    /// Output file (option: --output or -o)
    #[arg(short, long)]
    output: Option<PathBuf>,

    /// Print extra info (flag: --verbose or -v)
    #[arg(short, long)]
    verbose: bool,
}

fn main() {
    let args = Cli::parse();

    if args.verbose {
        println!("Input: {:?}", args.input);
    }

    match args.output {
        Some(path) => println!("Writing to {:?}", path),
        None       => println!("Writing to stdout"),
    }
}
$ cargo run -- input.txt -v --output out.txt
Input: "input.txt"
Writing to "out.txt"
short vs long

#[arg(short)] = use first letter of field name as -x flag. #[arg(long)] = use full field name as --field-name. You can override: #[arg(short = 'O', long = "out")].

Chapter 5

Options & Flags — Deep Dive

Counting flags (e.g. -vvv)

Use action = ArgAction::Count to count how many times a flag appears.

rust
use clap::{Parser, ArgAction};

#[derive(Parser)]
struct Cli {
    /// Verbosity (-v = info, -vv = debug, -vvv = trace)
    #[arg(short, long, action = ArgAction::Count)]
    verbose: u8,
}

fn main() {
    let args = Cli::parse();
    println!("verbosity level: {}", args.verbose);
}
$ ./myapp -vvv
verbosity level: 3

Negatable flags (--flag / --no-flag)

rust
#[derive(Parser)]
struct Cli {
    /// Enable colors (default: on). Use --no-color to disable.
    #[arg(long, default_value_t = true, action = clap::ArgAction::Set)]
    color: bool,
}
$ ./myapp --no-color
Chapter 6

Parsing into Rust Types

clap uses the ValueParser trait under the hood. For most standard types, it Just Works™ because they implement FromStr.

Rust typeCLI inputNotes
String--name fooAny text
u8, u32, i64, f64--port 8080Auto-parsed, error if invalid
boolflag present/absentSee flags section
PathBuf--input ./foo.txtOS path
Option<T>omit the flag entirelyNone if not provided
Vec<T>--tag a --tag bMultiple values
Custom enum (ValueEnum)--format jsonCovered in ValueEnum section
rust
use std::net::IpAddr;
use std::path::PathBuf;

#[derive(Parser, Debug)]
struct Cli {
    /// Port to listen on
    #[arg(short, long, default_value_t = 3000)]
    port: u16,

    /// Bind address
    #[arg(long, default_value = "127.0.0.1")]
    host: IpAddr,

    /// Config file path
    #[arg(short, long)]
    config: Option<PathBuf>,
}
default_value_t vs default_value

Use default_value_t = 3000 when your type is not a string (u16, bool, etc.). Use default_value = "text" for strings. Mixing them causes compile errors.

Chapter 7

Required & Optional Arguments

The type of your field determines whether the argument is required:

Field typeBehavior
String, u32, etc.Required — clap errors if missing
Option<T>Optional — None if not provided
bool (flag)Optional — false if not provided
Has default_valueOptional — uses default
rust
#[derive(Parser)]
struct Cli {
    /// Required: must be provided
    name: String,

    /// Optional: fine to omit
    #[arg(short, long)]
    nickname: Option<String>,

    /// Optional with default
    #[arg(short, long, default_value_t = 42)]
    count: u32,
}
Making an option required

You can force a named option to be required even if you'd normally use Option<T>: add #[arg(required = true)]. But usually just use T (non-option) instead.

Chapter 8

Default Values

rust
#[derive(Parser)]
struct Cli {
    /// String default
    #[arg(long, default_value = "world")]
    name: String,

    /// Numeric default — note _t suffix means "typed"
    #[arg(long, default_value_t = 8080)]
    port: u16,

    /// Boolean default
    #[arg(long, default_value_t = true)]
    color: bool,

    /// Computed default using a function
    #[arg(long, default_value_t = default_threads())]
    threads: usize,
}

fn default_threads() -> usize {
    std::thread::available_parallelism()
        .map(|n| n.get())
        .unwrap_or(4)
}
Chapter 9

Subcommands

Subcommands let you build CLIs like git where the first argument picks an operation (git add, git commit, git push). In clap, subcommands are enum variants.

rust src/main.rs 
use clap::{Parser, Subcommand};

#[derive(Parser)]
#[command(about = "A task manager")]
struct Cli {
    #[command(subcommand)]
    command: Commands,
}

#[derive(Subcommand)]
enum Commands {
    /// Add a new task
    Add {
        /// Task description
        description: String,
        /// Priority level
        #[arg(short, long, default_value_t = 1)]
        priority: u8,
    },
    /// List all tasks
    List {
        /// Show only done tasks
        #[arg(long)]
        done: bool,
    },
    /// Remove a task by ID
    Remove {
        id: u64,
    },
}

fn main() {
    let cli = Cli::parse();

    match cli.command {
        Commands::Add { description, priority } => {
            println!("Adding task: '{}' (priority {})", description, priority);
        }
        Commands::List { done } => {
            println!("Listing {} tasks", if done { "done" } else { "all" });
        }
        Commands::Remove { id } => {
            println!("Removing task {}", id);
        }
    }
}
$ ./tasks add "buy milk" --priority 2
Adding task: 'buy milk' (priority 2)

$ ./tasks list --done
Listing done tasks

$ ./tasks --help
A task manager
Usage: tasks <COMMAND>
Commands:
add Add a new task
list List all tasks
remove Remove a task by ID
Chapter 10

Required Subcommands & Optional Subcommands

By default, subcommands are required. Make them optional with Option<Commands>:

rust
#[derive(Parser)]
struct Cli {
    /// Optional: running with no subcommand shows status
    #[command(subcommand)]
    command: Option<Commands>,
}

fn main() {
    let cli = Cli::parse();
    match cli.command {
        Some(cmd) => run_command(cmd),
        None      => show_status(),
    }
}
Chapter 11

Global Arguments

A global argument is available to all subcommands. Add #[arg(global = true)] to the top-level struct.

rust
#[derive(Parser)]
struct Cli {
    /// Available in ALL subcommands
    #[arg(global = true, short, long)]
    verbose: bool,

    #[command(subcommand)]
    command: Commands,
}
$ ./myapp --verbose list ← global flag before subcommand
$ ./myapp list --verbose ← global flag after subcommand (both work!)
Chapter 12

Built-in Validation

Use #[arg(value_parser = ...)] to constrain accepted values.

rust
#[derive(Parser)]
struct Cli {
    /// Port must be 1024–65535
    #[arg(long, value_parser = clap::value_parser!(u16).range(1024..=65535))]
    port: u16,

    /// Number of workers: 1–16
    #[arg(long, value_parser = clap::value_parser!(u8).range(1..=16))]
    workers: u8,
}
$ ./server --port 80
error: invalid value '80' for '--port <PORT>': 80 is not in 1024..=65535
Chapter 13

Custom Validators

Write any function fn(&str) -> Result<T, E> and pass it to value_parser.

rust
fn parse_key_val(s: &str) -> Result<(String, String), String> {
    s.split_once('=')
     .map(|(k, v)| (k.to_string(), v.to_string()))
     .ok_or_else(|| format!("expected KEY=VALUE, got '{s}'"))
}

#[derive(Parser)]
struct Cli {
    /// Extra headers as KEY=VALUE
    #[arg(long, value_parser = parse_key_val)]
    header: Vec<(String, String)>,
}
$ ./myapp --header Content-Type=json --header X-Token=abc123
Chapter 14

Conflicts & Requires

Encode logical relationships between arguments so clap validates them automatically.

rust
#[derive(Parser)]
struct Cli {
    /// Quiet mode (conflicts with --verbose)
    #[arg(long, conflicts_with = "verbose")]
    quiet: bool,

    /// Verbose mode (conflicts with --quiet)
    #[arg(long, conflicts_with = "quiet")]
    verbose: bool,

    /// Output file (requires --format)
    #[arg(long, requires = "format")]
    output: Option<String>,

    /// Output format (required if --output is set)
    #[arg(long)]
    format: Option<String>,
}
$ ./myapp --quiet --verbose
error: the argument '--quiet' cannot be used with '--verbose'

$ ./myapp --output result.txt ← missing --format
error: the following required arguments were not provided: --format <FORMAT>
Chapter 15

ValueEnum — Restrict to Known Values

Use #[derive(ValueEnum)] on an enum to restrict an argument to a fixed set of values. clap generates the valid options list in help text automatically.

rust
use clap::{Parser, ValueEnum};

#[derive(ValueEnum, Clone, Debug)]
enum OutputFormat {
    Json,
    Toml,
    Yaml,
    #[value(name = "plain-text")]  // custom CLI name
    PlainText,
}

#[derive(Parser)]
struct Cli {
    /// Output format [json, toml, yaml, plain-text]
    #[arg(short, long, value_enum, default_value_t = OutputFormat::Json)]
    format: OutputFormat,
}

fn main() {
    let args = Cli::parse();
    match args.format {
        OutputFormat::Json      => println!("outputting JSON"),
        OutputFormat::Toml      => println!("outputting TOML"),
        OutputFormat::Yaml      => println!("outputting YAML"),
        OutputFormat::PlainText => println!("outputting plain text"),
    }
}
$ ./myapp --format toml
outputting TOML

$ ./myapp --format xml
error: invalid value 'xml' for '--format <FORMAT>'
[possible values: json, toml, yaml, plain-text]
Chapter 16

Environment Variables

Add features = ["env"] to your Cargo.toml, then use #[arg(env = "VAR_NAME")]. CLI flags take precedence over env vars.

toml Cargo.toml 
clap = { version = "4", features = ["derive", "env"] }
rust
#[derive(Parser)]
struct Cli {
    /// Database URL — reads $DATABASE_URL if not set on CLI
    #[arg(long, env = "DATABASE_URL")]
    database_url: String,

    /// API key — hides value in help output
    #[arg(long, env = "API_KEY", hide_env_values = true)]
    api_key: String,
}
$ DATABASE_URL=postgres://localhost/db ./myapp
# No --database-url needed — reads from env

$ ./myapp --database-url postgres://prod/db
# CLI flag overrides env var

Help output shows where a value came from:

--database-url <DATABASE_URL> [env: DATABASE_URL=postgres://localhost/db]
Chapter 17

Config Files (via clap + figment / config)

clap alone doesn't read config files — but it pairs well with figment or config-rs. The pattern: parse the config path with clap, then load it yourself.

rust Config file pattern 
use clap::Parser;
use std::path::PathBuf;

#[derive(Parser)]
struct Cli {
    /// Path to config file
    #[arg(long, default_value = "config.toml")]
    config: PathBuf,

    /// Override port from config
    #[arg(long)]
    port: Option<u16>,
}

fn main() {
    let cli = Cli::parse();
    let file_config = load_toml(&cli.config);

    // CLI takes precedence over file config
    let port = cli.port.unwrap_or(file_config.port);
    println!("Using port: {port}");
}
Typical precedence chain

CLI flags → environment variables → config file → hardcoded defaults. This is the standard Unix convention and what users expect.

Chapter 18

Multiple Values

Three ways to accept multiple values for one argument:

rust
#[derive(Parser)]
struct Cli {
    /// 1. Repeated flags: --tag a --tag b --tag c
    #[arg(long)]
    tag: Vec<String>,

    /// 2. Space-separated after flag: --file a.txt b.txt c.txt
    #[arg(long, num_args = 1..)]
    file: Vec<String>,

    /// 3. Exactly N values: --coords 10 20
    #[arg(long, num_args = 2)]
    coords: Vec<f64>,
}
$ ./myapp --tag rust --tag cli --tag cool
$ ./myapp --file a.txt b.txt c.txt
$ ./myapp --coords 12.5 -3.7
Chapter 19

Argument Groups

Argument groups let you say "exactly one of these must be provided" or "at most one of these can be provided" across multiple arguments.

rust
use clap::{Parser, ArgGroup};

#[derive(Parser)]
#[command(group(
    ArgGroup::new("auth")
        .required(true)       // one of these must be given
        .args(["api_key", "token"])
))]
struct Cli {
    /// Authenticate with API key
    #[arg(long)]
    api_key: Option<String>,

    /// Authenticate with bearer token
    #[arg(long)]
    token: Option<String>,
}
$ ./myapp
error: the following required arguments were not provided:
<--api-key <API_KEY>|--token <TOKEN>>

Custom Help & Version

rust
use clap::Parser;

#[derive(Parser)]
#[command(
    name = "myapp",
    version = "1.2.0",             // or env!("CARGO_PKG_VERSION")
    author = "Irshad <me@example.com>",
    about = "Does awesome things",
    long_about = "A longer description shown with --help but not -h",
    after_help = "EXAMPLES:\n  myapp --port 3000\n  myapp list --verbose",
)]
struct Cli {
    #[arg(short, long)]
    port: Option<u16>,
}

Use env!("CARGO_PKG_VERSION") for auto version

rust
#[command(version = env!("CARGO_PKG_VERSION"))]

This reads the version from Cargo.toml at compile time. You never need to update it manually.

#[arg(long, hide = true)]   // hidden from --help output

Chapter 21

Error Handling

Cli::parse() exits on error. If you want to handle errors yourself (e.g. in tests, or for custom error formatting), use Cli::try_parse().

rust
fn main() {
    let cli = Cli::try_parse().unwrap_or_else(|e| {
        // e.kind() lets you check the error type
        eprintln!("Error: {e}");
        std::process::exit(1);
    });

    // Or use the built-in pretty exit:
    let cli = Cli::try_parse().unwrap_or_else(|e| e.exit());
}

Testing your CLI

rust
#[test]
fn test_defaults() {
    let cli = Cli::parse_from(["myapp"]);  // inject args
    assert_eq!(cli.port, 8080);
}

#[test]
fn test_invalid_port() {
    let result = Cli::try_parse_from(["myapp", "--port", "99999"]);
    assert!(result.is_err());
}
Always test your CLI

Use parse_from and try_parse_from in unit tests. They accept anything that iterates over strings — no need for integration tests just to check argument parsing.

Real-world Patterns

Pattern 1: Git-style tool (global flags + subcommands)

rust
#[derive(Parser)]
#[command(name = "mytool", version, propagate_version = true)]
struct Cli {
    #[arg(global = true, long, action = ArgAction::Count)]
    verbose: u8,

    #[arg(global = true, long, default_value = "config.toml")]
    config: PathBuf,

    #[command(subcommand)]
    command: Commands,
}

Pattern 2: Server with env + CLI precedence

rust
#[derive(Parser)]
struct ServerCli {
    #[arg(long, env = "PORT", default_value_t = 8080)]
    port: u16,

    #[arg(long, env = "DATABASE_URL")]
    database_url: String,

    #[arg(long, env = "RUST_LOG", default_value = "info")]
    log_level: String,
}

Pattern 3: Pipe-friendly CLI (stdin or file)

rust
use std::io::{self, Read};
use std::path::PathBuf;

#[derive(Parser)]
struct Cli {
    /// Input file. Reads stdin if omitted.
    input: Option<PathBuf>,
}

fn main() {
    let cli = Cli::parse();
    let content = match cli.input {
        Some(path) => std::fs::read_to_string(path).unwrap(),
        None => {
            let mut buf = String::new();
            io::stdin().read_to_string(&mut buf).unwrap();
            buf
        }
    };
    // process content...
}
You're ready

These patterns cover 95% of real CLIs. The official docs at docs.rs/clap are excellent for edge cases — but you now have the mental model to read them without confusion.
